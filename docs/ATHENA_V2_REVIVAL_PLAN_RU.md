# Athena v2 Revival — regression-driven plan (RU)

> **Статус:** active track  
> **Дата перезапуска:** 2026-04-09

---

## 1. Новый статус `v2`

`Athena v2` больше не считается полностью закрытой веткой.

Теперь она возвращена как **вторая независимая линия разработки**:
- отдельно от `v3`;
- без смешивания router-first архитектуры;
- с фокусом на **regression-based восстановление рабочей логики**.

---

## 2. Базовая гипотеза

Есть высокая вероятность, что у `v2` был **конкретный момент деградации**:
- изменение label/features;
- drift в inference/train schema;
- переусложнение фильтров;
- неверная калибровка confidence / risk.

Задача — не "изобретать всё заново", а **найти точку поломки и восстановить то, что работало**.

---

## 3. Метод работы

Используем только **regression-driven** цикл:

1. зафиксировать reference baseline;
2. воспроизвести хороший/плохой режим на одинаковых окнах;
3. вносить по одной правке;
4. после каждой правки сравнивать:
   - число сделок,
   - return,
   - Sharpe,
   - PF,
   - side / hour / regime breakdown;
5. если правка не даёт подтверждения — откатываем/архивируем.

---

## 4. Ближайшие deliverables

1. Восстановить timeline: **когда именно `v2` пошла не туда**.
2. Собрать список regression checkpoints.
3. Прогнать базовый `v2` recheck в `data/results/v2/`.
4. Начать серию узких правок по одной гипотезе за раз.

---

## 5. Правило чистоты ветки

`v2-revival`:
- не тянет `RegimeRouter`;
- не тянет `quiet_mean_reversion_v1` и другие `v3` route-aware прототипы;
- допускает только те правки, которые улучшают `v2` в собственной логике.

---

## 6. Критерий успеха

Ветка `v2-revival` считается живой, если:
- improvement повторяется на strict OOS;
- результат не держится только на одной микровыборке;
- метрики не хуже архивного baseline и есть понятная причинно-следственная связь.
